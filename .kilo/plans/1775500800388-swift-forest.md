# Urban Company Cleaning Service Portal - Technical Plan

## Overview

Building an online house cleaning service portal with customer, provider (self-service marketplace), and admin roles. Features include service catalog & booking, reviews & ratings, provider dashboard, notifications, and payment stub.

## Technical Stack

- Next.js 16.1.7 (React 19)
- PostgreSQL with Prisma ORM
- BetterAuth for authentication
- Tailwind CSS 4.2.1
- TypeScript

## Database Schema Extensions

### New Models Needed:

1. **Service** - Cleaning services offered
2. **Booking** - Customer bookings
3. **Review** - Service reviews & ratings
4. **ProviderProfile** - Extended provider information
5. **Notification** - System notifications
6. **Payment** - Payment records (stub)

### Updated Models:

- **User** - Add role field (customer/provider/admin)
- **Session** - Already exists via BetterAuth
- **Account** - Already exists via BetterAuth

## Implementation Plan

### Phase 1: Foundation & Authentication (Days 1-2)

1. Extend Prisma schema with new models
2. Create authentication routes (login, register, logout)
3. Implement role-based access control middleware
4. Set up database migrations

### Phase 2: Core Features (Days 3-5)

1. **Service Catalog**
   - Service creation/management (provider/admin)
   - Service listing/browsing (customer)
   - Service filtering & search
2. **Booking System**
   - Availability checking
   - Booking creation workflow
   - Booking status management
3. **Provider Dashboard**
   - Provider registration/profile setup
   - Booking management interface
   - Earnings overview

### Phase 3: Additional Features (Days 6-7)

1. **Reviews & Ratings**
   - Review submission post-service
   - Rating display & aggregation
   - Review moderation (admin)
2. **Notifications**
   - Email notifications (booking confirmations, reminders)
   - In-app notification system
3. **Payment Stub**
   - Mock payment processing
   - Payment status tracking

### Phase 4: Polish & Testing (Days 8-9)

1. UI/UX refinements
2. Responsive design implementation
3. Error handling & loading states
4. Performance optimization
5. Comprehensive testing

## API Routes Structure

- `/api/auth/*` - BetterAuth endpoints
- `/api/services` - Service CRUD operations
- `/api/bookings` - Booking management
- `/api/reviews` - Review system
- `/api/providers` - Provider profile management
- `/api/notifications` - Notification endpoints
- `/api/payments` - Payment processing stub

## Component Architecture

### Pages:

- `app/page.tsx` - Landing page
- `app/services/page.tsx` - Service catalog
- `app/bookings/page.tsx` - User bookings
- `app/dashboard/page.tsx` - Role-based dashboard
- `app/profile/page.tsx` - User profile
- `app/login/page.tsx` - Authentication
- `app/register/page.tsx` - User registration

### Components:

- `components/services/` - Service cards, filters
- `components/bookings/` - Booking forms, lists
- `components/reviews/` - Rating display, review forms
- `components/layout/` - Header, footer, sidebar
- `components/ui/` - Reusable UI components (shadcn-based)

## Security Considerations

1. Role-based access control on all API routes
2. Input validation and sanitization
3. Rate limiting on public endpoints
4. Secure password handling via BetterAuth
5. CSRF protection
6. Data access restrictions (users only see their own data unless admin/provider)

## Scalability & Performance

1. Database indexing on frequently queried fields
2. Pagination for large datasets
3. Caching strategy for service catalog
4. Optimistic UI updates where appropriate
5. Lazy loading for images and components

## Development Workflow

1. Feature branching with descriptive names
2. Conventional commits
3. Pre-commit hooks for linting and formatting
4. PR reviews before merging to main
5. Automated testing setup (Jest/Vitest)
